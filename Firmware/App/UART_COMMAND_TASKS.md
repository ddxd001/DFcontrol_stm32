# UART 指令与任务对应文档（详细动作版）

本文档与当前 `Firmware/App/app_tasks.c` 对齐，用于现场调试时快速确认每条指令的动作顺序与参数。

## 1. 协议与运行约束

- 帧格式：`AA BB XX`
- 解析任务：`App_Task_10ms()`
- 串口：`USART1`（`DebugUart`）接收上述指令帧。
- **Linux 链路**：**`USART6`（UART6）** 使用 **`AA CC XX`** 三字节帧收发（收发逻辑见本文 **§9**）。
- 上电保护：前约 `1200ms` 触发请求会被清空
- 调度节拍：
  - 路口检测任务：`10ms`（100Hz）
  - 主状态机任务：`100ms`（等待时间以该节拍计数）

## 4. 各任务详细动作

### Q1（`AA BB 02`）

1. 蜂鸣一次后启动。
2. 前进 `0.08m`。
3. 循环 4 轮：左转 `90°` -> 直走。
4. 前 3 轮直走 `1.0m`，第 4 轮直走 `0.92m`。
5. 最后停车并蜂鸣。

### Q2-1（`AA BB 03`）

1. 蜂鸣一次后启动。
2. 前进 `0.08m`。
3. 循环 3 轮：左转 `90°` -> 直走 `1.0m`。
4. 调头 `180°`。
5. 返程 3 段：`1.0m` + `1.0m` + `0.9m`。
6. 停车并蜂鸣。

### Q2-2（`AA BB 04`）

与 Q2-1 相同，仅首段转向为右转 `90°`。

### Q3-1（`AA BB 05`）

1. 总体结构与 Q1 相同，按轮次运行。
2. 第 2 次直走（`s_round == 1`）替换为“测距流程”（与 TEST1 同构）：
   - 读取灰度偏移并做小角度修正（若偏差足够大才修正）。
   - 修正后稳定等待 `0.3s`。
   - 匀速前进 `0.2m`（速度10）。
   - 反向转回修正角，稳定等待 `0.2s`。
   - 校准直行后等待 `0.6s`（无蜂鸣，直接进测量段）。
   - 匀速前进 `0.4m`（检测段，速度5，开启路口检测与测距）。
   - 稳定等待 `2.2s`（检测窗口）。
   - 自适应前进 `0.4m`，结束该替换段。
3. 替换段结束后，额外等待 `1.0s` 再回到 Q3-1 主循环继续，最终停车并蜂鸣。

### Q3-2（`AA BB 06`）

1. 蜂鸣一次后启动。
2. 自适应前进 `0.08m`。
3. 执行 2 次循环：`右转90° + 自适应直走1.0m`。
4. 再右转 `90°`，进入与 TEST1 同构的测距流程。
5. 测距流程结束后，再执行：`右转90° + 自适应直走1.0m`。
6. 稳定等待后停车并蜂鸣。

### Q4-V（`AA BB 0B`）

1. 蜂鸣一次后启动。
2. 前进 `0.08m`。
3. 左转 `90°`，稳定等待。
4. 前进 `0.55m`，完成后经 `UART6` 发送 `AA CC 08`。
5. 等待 `4.0s`。
6. 前进 `0.5m`，随后稳定等待约 `1.4s`。
7. 左转 `90°`（同时下发步进电机右转90°），稳定等待后经 `UART6` 发送 `AA CC 07`。
8. 前进 `0.92m`（自适应直走速度15）。
9. 稳定等待约 `3.0s` 后，经 `UART6` 发送 `AA CC 00` 询问下一步。
10. 等待 `UART6` 返回 `AA CC XX`：
    - `EE`：蜂鸣三次后结束（空闲）。
    - `01`：执行 TEST3-4。
    - `02`：执行 TEST3-1。
    - `03`：执行 TEST3-3。
    - `04`：执行 TEST3-2。

### Q4-1（`AA BB 07`）

1. 先执行完整 Q4-V 流程。
2. Q4-V 结束后，接续执行 TEST3-2 动作：
   - 左转 `116.5°`
   - 前进 `0.745m`
3. 稳定等待后停车并蜂鸣。

### TEST1（`AA BB D0`）

1. 蜂鸣一次后启动。
2. 检测前修正段：
   - 灰度偏移修正（可选）。
   - 修正稳定等待 `0.3s`（与 Q3-1 替换段一致）。
   - 匀速前进 `0.2m`（速度10）。
   - 反向转回修正角并稳定等待 `0.2s`（与 Q3-1 替换段一致）。
3. 测距主段：
   - （无中间蜂鸣，直接进入测量段）
   - 匀速前进 `0.4m`（速度5，开启路口检测与测距）。
   - 稳定等待 `2.2s`（检测窗口）。
4. 收尾段：
   - 自适应前进 `0.4m`。
   - 停车并蜂鸣。

### TEST1-REC（`AA BB DA`）

与 **TEST1 相同的行驶路径与计时**。区别在测距主段：

- **不**做实时路口计数；`10ms` 任务内对灰度只做 `GwGray_ReadDigitalUpdate`，将 `digital_inv` 与 `HAL_GetTick` 写入缓冲（容量 **320** 点，覆盖约 `3.2s@100Hz`）。
- 离开检测窗口后、**计算 `l1`/`l2` 之前**：经 **`USART1`** 以 **VOFA+ FireWater** 风格发原始采样（逗号分隔数值，行尾为 **CRLF**）：
  - 首行：采样个数 — `t1r_m:<cnt>`
  - 此后每行一点：`t1r:<idx>,<tick_ms>,<inv>`（`inv` 为 `digital_inv` 的十进制）
- 随后在**进入自适应 `0.4m` 之前**，对整段缓冲离线运行与 **§5** 相同的防抖（`APP_CROSS_ON_CONFIRM_FRAMES` / `APP_CROSS_OFF_CONFIRM_FRAMES`）与 `GwGray_IsCrossByDigitalInv(…, APP_GRAY_CROSS_MIN_ACTIVE_BITS)`，提取至多 **3** 个路口时刻，再按与 TEST1 相同的 `dt_ms`→`cm` 公式填入测距日志，经 **`UART4`** 下发 `l1` / `l2`。
- 检测窗内**无**逐路口蜂鸣（ analysis 完成后照常走收尾停车蜂鸣）。
- PC 端可用仓库 **`analysis/parse_test1rec.py`** 解析 USART1 保存的 FireWater 文本，复现 `l1`/`l2`（可选 **`--csv`**）；**`--plot`** 会绘制八路 **digital_inv** 热力图、popcount 与防抖路口曲线，**`--plot-save xxx.png`** 可保存组图。

## 5. 路口检测与测距（TEST1 + TEST1-REC + Q3-1替换段）

- 触发窗口：仅在检测段（匀速 `0.4m` + 等待窗口）内有效。
- **TEST1 / Q3-1 替换段**：实时流程；**TEST1-REC**：同窗内仅采样，段末离线使用相同判定与防抖（`GwGray_IsCrossByDigitalInv` + 连续帧确认），再算 `l1/l2`。
- 判定方法（实时或离线）：`GwGray_ReadAndDetectCross()` 或等价的 `digital_inv` + `GwGray_IsCrossByDigitalInv()`，阈值 `APP_GRAY_CROSS_MIN_ACTIVE_BITS = 3`。
- 时间层连续帧确认（100Hz）：
  - 连续 `4` 帧判定为路口才确认进入（约40ms，`APP_CROSS_ON_CONFIRM_FRAMES`）
  - 连续 `3` 帧判定非路口才确认退出（约30ms，`APP_CROSS_OFF_CONFIRM_FRAMES`）
- 计数方式：锁存上升沿计数（`latched`），每次新路口蜂鸣一次。
- 路口上限：`APP_MEAS_EXPECT_CROSS_COUNT = 3`，达到后停止后续检测。
- 缓冲策略：检测期间仍将路口与间距写入缓冲区，检测段结束时仅用于 **`UART4` 蓝牙** 下发 `l1/l2`，**不再通过 `USART1` 输出文本日志**（**例外**：**TEST1-REC** 在算距前经 `USART1` 发 FireWater 采样流，见 §4.1）。
- 当前测距公式（相邻路口）：
  - `距离(cm,保留1位) = dt_ms * 0.15 * 0.798 * 100 * 10 / 1000`
  - 整数实现：`dist_cm_x10 = (dt_ms * 1197 + 5000) / 10000`
- 距离钳位：`1.0cm ~ 5.0cm`（超出范围按边界值输出）。
- 蓝牙发送（`UART4`）：
  - 发送时机：测距段结束并刷新日志时
  - 发送格式（先空行）：
    - `l1 = 数据1cm`
    - `l2 = 数据2cm`
  - 若缺失项则发送默认值 `2.5cm`

## 6. 步进电机（`08/09/0A`）

- 驱动：`StepperMotorDrv`
- 串口：`USART2`（`PA2 TX / PD6 RX`）
- 指令行为：
  - `08`：使能
  - `09`：失能
  - `0A`：左转90°（相对位移，`pulse = -800`）
- **说明**：步进使能/失能 **仅** 通过 **`AA BB 08` / `AA BB 09`**；**`PD4` 用于 TEST1-GYCK（灰度自检）**，见 **§7.3**。
- 速度参数：`100`

## 7. 串口输出

- `USART1`：任务指令接收（`AA BB XX`，见指令总表），**不测距文本日志**；**TEST1-REC** 仅在检测窗结束后输出 **FireWater** 采样流（见 §4.1）。
- **`USART6`**：与 Linux 交互的 **`AA CC xx`** 帧，收发说明见 **§9**。
- **`UART4`（蓝牙）**：
  - 测距：`l1 = ...cm`、`l2 = ...cm`（格式同前）。
  - 灰度 **10 Hz 看门狗判离线**（连续 `Ping` 失败达阈值）时，**仅在线→离线瞬间**发送一行：`\r\nGRAY:OFF\r\n`。
  - **TEST6**（`AA BB D8`）：见 **§7.1**（`HAL_UART_Transmit` → **`huart4`**）。
  - **TEST7**（`AA BB D9`，见 **§7.2**）；**TEST1-GYCK**（`AA BB DB` 或 **PD4**，见 **§7.3**）。

### 7.1 TEST6（`AA BB D8`，灰度 / I2C 验证）

- **用途**：验证灰度**热插拔**后 **`s_gray_sensor_inited`（看门狗）、`GwGray_Ping`、数字量读、`hi2c1` 状态**是否一致，用于排查「拔掉再插后路口不识别」是否与 I2C 卡死有关。
- **动作**：蜂鸣一次 → 经 **`UART4`（`HAL_UART_Transmit(&huart4, …)`，`50 ms` 超时）** 先发一行横幅说明字段 → 再起动 **120 秒 × 1 Hz** 采样（与 `App_GraySensorWatchdog10Hz()` 同运行于 `App_Task_100ms`，顺序为：先看门狗再起动本测试逻辑）。
- **样例行格式**（行首 `\r\n`，与 `GRAY:OFF` / 测距下发风格一致）：
  - `\r\nT6 ms=<tick> init=<0|1> png=<0|1> dg=<0|1> st=<hh> e0=<hh> e1=<hh> di=<hh>\r\n`
  - **`init`**：`App_GraySensorWatchdog10Hz` 维护的在线标志（采样前瞬间副本）。
  - **`png` / `dg`**：本秒内依次调用 `GwGray_Ping`、`GwGray_ReadDigitalUpdate` 的成败（`1`=成功）。
  - **`st`**：`hi2c1.State` 的低 8 位（十六进制），对照 `stm32f4xx_hal_i2c.h` 中 `HAL_I2C_StateTypeDef`。
  - **`e0`**：Ping / 数字量读**之前**的 `HAL_I2C_GetError(&hi2c1)` 低 8 位；**`e1`**：**之后**的低 8 位（对照 `HAL_I2C_ERROR_*`）。
  - **`di`**：采样后 **`s_gray_sensor.digital_inv`**（感为寄存器读后缓存）。
- **再次触发**：可再次发送 `AA BB D8`；新会话重设计数与节拍（上一场未跑满 120 s 时由新触发覆盖）。
- **上电保护**：与其它任务相同，约前 **1200 ms** 内忽略 `D8`。

### 7.2 TEST7（路口上升沿蜂鸣）

- **用途**：验证「是否判到路口」：与测距段相同判据（**`GwGray_ReadAndDetectCross`**，阈值 **`APP_GRAY_CROSS_MIN_ACTIVE_BITS`**），在数字量**从非路口 → 路口**的上升沿 **`BuzzerDrv_Beep(80)`** 一次；停留在路口上不连续蜂鸣。
- **开关**：仅 **`AA BB D9`**（与 **`App_Test7_UserToggle()`** 等价）。每收到一次切换监听 **开/关**。**打开时再蜂鸣 1 次**作提示；**关闭时无额外蜂鸣**。
- **条件**：须 **`s_gray_sensor_inited != 0`**，否则不响。
- **调度**：于 **`App_Task_10ms`** 调用，顺序在 **`App_ProcessCrossDetect10ms` 之后**；与测距窗同时启用时同一周期可能对灰度多读一次 I²C。
- **透传**：USART1 ↔ UART5 透传开启时 **不解析 `D9`**；TEST7 需关闭透传或使用其它调试手段。热插拔后快速确认 I²C 可读请用 **§7.3（PD4）** 或 **TEST6（`D8`）**。

### 7.3 TEST1-GYCK（灰度可读性，PD4 / `AA BB DB`）

- **用途**：不依赖 **`s_gray_sensor_inited`** 门控，直接 **`GwGray_Ping`** → **`GwGray_ReadDigitalUpdate`**：判断能否从模块拿到数字量。
- **反馈**：
  - **两步均成功**：**蜂鸣 1 次**（`BuzzerDrv_Beep(80)`）。
  - **任一步失败**：**蜂鸣 2 次**（与 TEST5 失败相同的**队列间隔蜂鸣**，`s_pending_beeps = 2`）。
- **触发**：
  - **PD4**：每按一次执行一次检测（**不受 USART1 透传影响**，逻辑在透传判定之前执行）。
  - **`AA BB DB`**：与 PD4 **相同动作**（仅当 **未** 处于 USART1↔UART5 透传、串口正常解析时有效）。
- **与 TEST6 区别**：本项**无 UART 文本上报**，仅听蜂鸣次数即可；需寄存器级诊断仍用 **`AA BB D8`**。

## 8. 按键（`PD3` / `PD4` / `PD5`，`App_Task_10ms`）

- **PD3**
  - 第 1 次按下：**UART6** 发送 **`AA CC DD`**（仅当发送成功后才进入下一阶段）。
  - 收到 **UART6** **`AA CC BB`**：**蜂鸣一次**（自本次按下 DD 后起，在未再次按下 PD3 前，仅第一次 BB 触发蜂鸣）。
  - 第 2 次按下：启动 **Q4-V**（与 `AA BB 0B` 相同）。
- **PD4**：**TEST1-GYCK**，灰度 **Ping + 读数字量**（成功 **1** 声、失败 **2** 声；见 **§7.3**）。
- **PD5**：底盘航向锁开关切换（成功后蜂鸣一次）。
- **上电约 `1200ms` 保护期**内按键事件无效，`PD3` 握手相位会复位。

## 9. USART6 — 连接 Linux（`AA CC` 协议）

本节描述 **UART6** 在固件中的**驱动方式**，以及 **`App_Task_10ms()`** 内对 **`AA CC xx`** 的**收发与语义**。与 **`USART1` 的 `AA BB XX`** 彼此独立。

### 9.1 硬件与链路参数（Cube / `MX_USART6_UART_Init`）

- **接口**：USART6，`115200`，`8N1`
- **引脚**：`PC6` = TX，`PC7` = RX（以工程 `ioc` / `MX_USART6_UART_Init` 为准）

### 9.2 底层收发（`Firmware/Drivers/debug_uart.c`）

| 方向 | 实现 |
|---|---|
| **RX** | **`HAL_UART_Receive_IT`** 单字节中断接收，写入 **`USART6` 环形缓冲**（在 **`DebugUart_Init()`** 内对 **`huart6`** 调用 **`HAL_UART_Receive_IT`** 启动）。 |
| **TX** | **`DebugUart6_Send`** → **`HAL_UART_Transmit`**（阻塞发送，超时由调用方毫秒参数指定）。 |
| **应用读取** | **`DebugUart6_ReadByte`**：`App_Task_10ms` 中轮询读出，非阻塞。 |

### 9.3 帧格式与对齐方式

- **定长 3 字节**：**`AA CC xx`**（`xx` 为命令字节）。
- 解析采用**滑动三字节日检**：每收到一字节即用「前两字节 + 当前字节」判断是否匹配某一种 `AA CC xx`，可多帧首尾相接，容错错位。

### 9.4 MCU → Linux（MCU 发出的帧）

| 帧 | 触发条件 | 超时 / 备注 |
|---|---|---|
| **`AA CC DD`** | **PD3 第一次按下**，且 **`DebugUart6_Send` 成功后**进入下一阶段握手 | TX 超时 `50 ms`，失败可再次按 PD3 重发 |
| **`AA CC 08`** | **Q4-V**：**第 4 步完成**（自适应前进 `0.55m` 指令下发成功）后立即发送（状态 **`126U`**） | TX `50 ms`；发送失败则在下一 **`100ms`** 状态周期重试该步 |
| **`AA CC 07`** | **Q4-V**：**第 7 步完成**（再左转 `90°` + 转向稳定等待结束）后发送（状态 **`29U`**） | 同上 |
| **`AA CC 00`** | **Q4-V 纯任务**末段：`0.92m` 直行及末段稳定等待完成后，询问 Linux 下一步（状态 **`27U`**） | 仅 **`AA BB 0B`** 路径；**Q4-1**（`AA BB 07`）在此前切入接续段，不进本询问 |
| **`AA CC 05`** | **`AA BB D7`（TEST5）**：每次触发发送一帧探测 | TX 超时 `20 ms` |
| **`AA CC 06`** | 收到 **`AA CC 06`** 时的**回声**：再发 **`AA CC 06`**（见 §9.5） | — |

### 9.5 Linux → MCU（MCU 解析并动作的帧）

| 帧 | 生效条件概要 | MCU 动作 |
|---|---|---|
| **`AA CC 06`** | 任意 | 立即 **回发 `AA CC 06`** |
| **`AA CC BB`** | **PD3 握手相位**为「已发 **`AA CC DD`**、尚未第二次按 PD3」 | **蜂鸣 1 次**；同一相位内仅响应**第一次 `BB`** |
| **`AA CC 05`** | **TEST5** 已发探测且在 **1000 ms** 窗内收到 | 判 **连接成功** → 随后 **蜂鸣 1 次**（队列 `s_pending_beeps`）；超时或发送失败等为 **蜂鸣 2 次**（失败） |
| **`AA CC EE` / `01`～`04`** | **仅当** Q4-V 已发 **`AA CC 00`** 且处于等待决策（状态 **`28U`**，`s_q4v_uart6_wait_resp != 0`） | 读第三字节：**`EE` 三连蜂鸣停走**；**`01`～`04`** 跳转 **TEST3-4 / TEST3-1 / TEST3-3 / TEST3-2**；其它码停到空闲 |

**说明**：接收到 **`AA CC 05`** 时，代码会清零内部「UART6 等待应答」标志；除 **TEST5** 外固件不会自发周期性发 **`05`**，一般互不干扰。

### 9.6 Q4-V：对 `AA CC 00` 的决策应答（第三字节）

| 应答 **`AA CC xx`**（`xx`） | MCU 后续 |
|---|---|
| **`EE`** | 蜂鸣 **3** 次，回空闲 |
| **`01`** | 启动 **TEST3-4** |
| **`02`** | 启动 **TEST3-1** |
| **`03`** | 启动 **TEST3-3** |
| **`04`** | 启动 **TEST3-2** |
