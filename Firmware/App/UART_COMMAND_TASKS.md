# UART 指令与任务对应文档（详细动作版）

本文档与当前 `Firmware/App/app_tasks.c` 对齐，用于现场调试时快速确认每条指令的动作顺序与参数。

## 1. 协议与运行约束

- 帧格式：`AA BB XX`
- 解析任务：`App_Task_10ms()`
- 串口：`USART1`（`DebugUart`）接收上述指令帧。
- **Linux 链路**：**`USART6`（UART6）** 使用 **`AA CC XX`** 三字节帧收发（收发逻辑见本文 **§9**）。
- 上电保护：前约 `1200ms` 触发请求会被清空
- 调度节拍：
  - 路口检测任务：`10ms`（100Hz）
  - 主状态机任务：`100ms`（等待时间以该节拍计数）

## 4. 各任务详细动作

### Q1（`AA BB 02`）

1. 蜂鸣一次后启动。
2. 前进 `0.08m`。
3. 循环 4 轮：**左转 `90°`**（**`r_max = 1500`**）后 **`state 4`** 稳定约 **`0.2s`（2 ticks）** → **直走**。
4. 前 3 轮直走 `1.0m`，第 4 轮直走 `0.94m`（在 `0.92m` 基础上多 `0.02m`）。
5. 最后停车并蜂鸣。

### Q2-1（`AA BB 03`）

`s_mode = 2`。直走（含收尾后退）均用 **`DflinkChassis_SendAdaptConstPMove`**，速度 **`move_speed_mps100`（文档 §3：速度30）**；±90° 用 **`rot_vmax_90 = 1500`**，180° 调头用 **`rot_vmax_u_turn = 1500`**。等待节拍为 **`App_Task_100ms`** 的 `s_wait_ticks`（1 tick = **0.1 s**）。

**去程（3 轮「转向 + 直走」）**

1. 蜂鸣一次，进入状态机 **`state 1`**。  
2. **前置短移**：前进 **`0.08m`**，随后 **`state 2`** 稳定等待约 **0.3 s**（**3** ticks）。  
3. **第 1 轮** **`state 3`～`6`**：**左转 `90°`** → **`state 4`** 转向稳定约 **0.6 s**（**6** ticks）→ **直走 `0.98m`** → **`state 6`** 行进稳定约 **1.4 s**（**14** ticks）→`s_round` 加 1。  
4. **第 2 轮**：再次 **`state 3`～`6`**：**左转 `90°`** → 等 **0.6 s** → **直走 `1.0m`**（**Q2-2** 为 **`0.95m`**，少 **`0.05m`**）→ 等 **1.4 s**（`s_round` 再增）。  
5. **第 3 轮**：**左转 `90°`** → 等 **0.6 s** → **直走 `0.90m`**（**Q2-1**；**Q2-2** 见下表为 **`0.95m`**）→ 等 **1.4 s** 后 `s_round = 3`，进入 **`state 7`**。

**调头**

6. **`state 7`**：**左转 `180°`**（`rot_u_turn`）。  
7. **`state 8`**：稳定等待约 **0.9 s**（**9** ticks），进入返程 **`state 9`**。此时 `s_ret_seg = 0`。

**返程（3 段直走，两段间各转 90°）**

8. **返程第 1 段**（调头后第一段直走）：**Q2-1** **`0.91m`**、**Q2-2** **`0.94m`** → **`state 10`** 稳定约 **1.5 s**（**15** ticks，较第 2/3 段 **+100ms**）→ **`state 11`**：返程拐角 **Q2-1 右转 `90°` / Q2-2 左转 `90°`** → **`state 14`** 稳定约 **0.4 s**（**4** ticks）→`s_ret_seg = 1`，回到 **`state 9`**。  
9. **返程第 2 段**（倒数第二个直走）：直走 **`0.99m`**（较 **`0.94m` 多 `0.05m`**）→ **`state 10`** 稳定约 **1.7 s**（**17** ticks，较默认 **1.4 s** **+300ms**）→ **`state 11`**：返程拐角 **Q2-1 右转 `90°` / Q2-2 左转 `90°`** → 等 **0.4 s** →`s_ret_seg = 2`，回到 **`state 9`**。
10. **返程第 3 段**（最后一个直走）：**Q2-1** **`0.95m`**、**Q2-2** **`0.98m`**（**Q2-1** 较 **`0.98m` 少 `0.03m`**）→ **`state 10`** 等 **1.4 s** 后进入 **`state 127`**（不再经 **`state 11`**）。

**收尾**

11. **`state 127`**：**左转 `90°`**（任务结束朝向）。  
12. **`state 128`**：转向稳定约 **0.6 s**（**6** ticks）。  
13. **`state 129`**：收尾后退，**Q2-1 `0.05m` / Q2-2 `0.11m`**（**`DflinkChassis_SendAdaptConstPMove`**，`Py` 负向）。
14. **`state 130`**：稳定约 **1.4 s**（**14** ticks），**停车并蜂鸣**（空闲）。

### Q2-2（`AA BB 04`）

`s_mode = 3`。与 **Q2-1** 共用同一状态机，差异仅在于：

| 步骤含义 | Q2-1（`03`） | Q2-2（`04`） |
|---|---|---|
| 去程每轮起手的转弯（**`state 3`**） | **左转 `90°`** | **右转 `90°`** |
| 去程第 1 轮直走（**`case 5`**，`s_round == 0`） | **`0.98m`** | **`1.0m`** |
| 去程第 2 轮直走（**`case 5`**，`s_round == 1`） | **`1.0m`** | **`0.95m`**（少 **`0.05m`**） |
| 去程第 3 轮直走（**`case 5`**，`s_round == 2`，调头前末段） | **`0.90m`** | **`0.95m`** |
| 调头后第一段直走（**`case 9`**，`s_ret_seg == 0`） | **`0.91m`** | **`0.94m`** |
| 返程第 2 段直走（**`case 9`**，`s_ret_seg == 1`，倒数第二段） | **`0.99m`** | **`0.99m`** |
| 返程末段直走（**`case 9`**，`s_ret_seg >= 2`） | **`0.95m`** | **`0.98m`** |
| 返程全部结束（**`state 127`**） | **左转 `90°`** | **右转 `90°`** |
| 收尾后退（**`state 129`**） | **`0.05m`** | **`0.11m`** |

其余距离、等待时间、段长：**Q2-1** 去程第 1 轮 **`0.98m`**、第 2 轮 **`1.0m`**、第 3 轮 **`0.90m`**；返程 **`0.91 + 0.99 + 0.95m`**。**Q2-2** 去程第 1 轮 **`1.0m`**、第 2 轮 **`0.95m`**、第 3 轮 **`0.95m`**；返程 **`0.94 + 0.99 + 0.98m`**。速度档与 **Q2-1** 相同。

### Q3-1（`AA BB 05`）

1. 总体结构与 **Q1** 相同，按轮次运行（**±90° 转向后稳定**见 §3：通常 **0.4s**，将进入测距替换段前、第二轮起手的那次为 **0s**）。
2. 第 2 次直走（`s_round == 1`）替换为“测距流程”（与 TEST1 同构）：
   - 读取灰度偏移并做小角度修正（若偏差足够大才修正）。
   - 修正后稳定等待 `0.3s`。
   - 匀速前进 `0.25m`（速度12）。
   - 反向转回修正角，稳定等待 `0.2s`。
   - 校准直行后等待 `0.6s`（无蜂鸣，直接进测量段）。
   - 匀速前进 `0.4m`（检测段，速度5，开启路口检测与测距）。
   - 稳定等待 `2.2s`（检测窗口）。
   - 自适应前进 `0.4m`，结束该替换段。
3. 替换段结束后，额外等待 `1.0s` 再回到 Q3-1 主循环继续；接续的**首次** ±90°（**`state 4`、`s_round == 2`**）稳定 **`0.2s`**（较常规 **`0.4s`** 少 **`200ms`**），最后一轮直走 **`0.94m`**，最终停车并蜂鸣。

### Q3-2（`AA BB 06`）

`s_mode = 5`。节拍均为 **`App_Task_100ms`**（文中 **tick** 若无说明即为该节拍，**1 tick = 0.1 s**）。

**运动与转弯参数**

- 自适应直走：**`DflinkChassis_SendAdaptConstPMove`**，**`move_speed_mps100`（§3：速度30）**。
- 右转 90°：**`DflinkChassis_SendRotation`**，**`rot_right_90`**，**`r_max`** 为 **`rot_vmax_90`（1500）**；**进入测距前**（`state 98`）与**测距结束后**（`state 101`）源码使用 **`rot_vmax_q3_case2_slow`**，**数值与 1500 相同**。

**详细步骤（与 `app_tasks.c` 状态号对齐）**

1. **启动**（`s_q3_case2_start_req`）：蜂鸣一次；**`s_mode = 5`**，**`s_state = 1`**，**`s_round = 0`**。  
   **注意**：不走路径 **Q1/Q2** 的通用「**`state 1` 前置 0.08 m**」；**Q3-2** 在 **`state 1`** 直接进入 **`state 92`**。

2. **`state 92`**：自适应前进 **`0.1m`**（`move_py_q3_case2_start_m21739`）。  
3. **`state 93`**：稳定等待 **`0.4s`**（**4 ticks**）。

4. **`state 94`**（循环入口）：若 **`s_round >= 2`** → **`state 98`**（测距前右转）；否则 → **`state 95`**。

5. **共 2 轮**「右转 90° + 自适应 1.0 m」（**`s_round == 0` 与 `s_round == 1` 各执行一整轮**）：  
   - **`state 95`**：右转 **90°**。  
   - **`state 96`**：转向稳定 —— **第 1 轮**（**`s_round == 0`**）**`0.3s`（3 ticks，`wait_ticks_q3_case2_first_turn`）**；**第 2 轮**（**`s_round == 1`**）**`0.5s`（5 ticks，`wait_ticks_q3_case2_other_turn`，较原先 **`0.3s` 多 `200ms`**）。  
   - **`state 97`**：自适应直走 **`1.0m`**。  
   - **`state 107`**：直走后稳定 **`1.4s`（14 ticks，`wait_ticks_default`）**；**`s_round++`**，回到 **`state 94`**。

6. **`state 98`**：再右转 **90°**（测距流程前最后一次右转）。  
7. **`state 99`**：稳定 **`0.5s`（5 ticks，`wait_ticks_q3_case2_other_turn`，较原先 **`0.3s` 多 `200ms`**）**。  
8. **`state 30`**：进入与 **TEST1** 完全同构的测距流程（灰度可选修正、**`0.25m` 校准直走速度12**、**`0.4m` 检测段速度5**、**`2.2s` 检测窗**、收尾自适应 **`0.4m`** 等），动作细节见本节下方 **「测距子流程」** 与 **§4 TEST1**、**§5**。

9. **`state 38`**（测距收尾段结束后的等待）：当 **`s_mode == 5`** 时**不停车**；稳定 **`1.2s`（12 ticks）** 后 → **`state 101`**（独立 **TEST1** 此处为 **`1.4s` 后停车并蜂鸣**）。

10. **`state 101`**：右转 **90°**。  
11. **`state 102`**：稳定 **`0.3s`（3 ticks，`wait_ticks_q3_case2_after_measure_90`，较原先 **`0.1s` 多 `200ms`**）**。  
12. **`state 103`**：自适应直走 **`0.90m`**（`move_py_ret_last_m21739`）。
13. **`state 104`**：稳定 **`1.4s`（14 ticks）** → **`App_Motion_StopAndBeep`**（停车并蜂鸣）。

**测距子流程（`state 30`～`38`，与 TEST1 相同）**

- **`30` → `42/43`**：可选灰度角修正；修正则 **`42`** 等 **`0.3s`** 再 **`43`**。  
- **`43`**：匀速 **`0.25m`**（**`move_speed_measure_calib_straight_mps100`，速度12**）。  
- **`31`**：校准后 **`0.6s`**（**6 ticks**，**`wait_ticks_pre_detect`**）；若有修正则 **`44`→`45`** 反向转回，**`45`** 等 **`0.2s`** 后 **`33`**。  
- **`33`**：匀速检测段 **`0.4m`**（**速度5**），开启路口检测与测距逻辑。

## 5. 路口检测与测距（TEST1 + TEST1-REC + Q3-1替换段）

- 触发窗口：仅在检测段（匀速 `0.4m` + 等待窗口）内有效。
- **TEST1 / Q3-1 替换段**：实时流程；**TEST1-REC**：同窗内仅采样，段末离线使用相同判定与防抖（`GwGray_IsCrossByDigitalInv` + 连续帧确认），再算 `l1/l2`。
- 判定方法（实时或离线）：`GwGray_ReadAndDetectCross()` 或等价的 `digital_inv` + `GwGray_IsCrossByDigitalInv()`，阈值 `APP_GRAY_CROSS_MIN_ACTIVE_BITS = 3`。
- 时间层连续帧确认（100Hz）：
  - 连续 `4` 帧判定为路口才确认进入（约40ms，`APP_CROSS_ON_CONFIRM_FRAMES`）
  - 连续 `3` 帧判定非路口才确认退出（约30ms，`APP_CROSS_OFF_CONFIRM_FRAMES`）
- 计数方式：锁存上升沿计数（`latched`），每次新路口蜂鸣一次。
- 路口上限：`APP_MEAS_EXPECT_CROSS_COUNT = 3`，达到后停止后续检测。
- 缓冲策略：检测期间仍将路口与间距写入缓冲区，检测段结束时仅用于 **`UART4` 蓝牙** 下发 `l1/l2`，**不再通过 `USART1` 输出文本日志**（**例外**：**TEST1-REC** 在算距前经 `USART1` 发 FireWater 采样流，见 §4.1）。
- 当前测距公式（相邻路口）：**dt_ms** 为相邻路口时间间隔（毫秒）。先取 **M = dt_ms × 0.15**，再得实际距离 **D**（厘米）
  - **D = 0.106986 × M − 1.633710** = **0.0160479 × dt_ms − 1.633710**
  - 蓝牙 **`dist_cm_x10`**（0.1 cm）：`round(10 × D)`，固件等价 **(160479 × dt_ms − 16337100 + 500000) / 1000000**，**≤0** 时按 **0**
  - 系数见 **`Firmware/Config/fw_config.h`** 中 **`APP_CROSS_DIST_LIN_*`**
- 蓝牙发送（`UART4`）：
  - 发送时机：测距段结束并刷新日志时
  - 发送格式（先空行）：
    - `l1 = 数据1cm`
    - `l2 = 数据2cm`
  - 若缺失项则发送默认值 `2.5cm`

## 6. 步进电机（`0C/0D/0E`）

- 驱动：`StepperMotorDrv`
- 串口：`USART2`（`PA2 TX / PD6 RX`）
- 指令行为：
  - `0C`：使能
  - `0D`：失能
  - `0E`：左转90°（相对位移，`pulse = -800`）
- **说明**：步进 **使能/失能** 可通过 **`AA BB 0C` / `AA BB 0D`** 与 **PD3**（与串口共用同一内部状态，**`HAL_OK` 才翻转**）；**`PD4` 用于 TEST1-GYCK**，见 **§7.3**。**PD3** 行为见 **§8**。
- 速度参数：`100`

## 7. 串口输出

- `USART1`：任务指令接收（`AA BB XX`，见指令总表），**不测距文本日志**；**TEST1-REC** 仅在检测窗结束后输出 **FireWater** 采样流（见 §4.1）。
- **`USART6`**：与 Linux 交互的 **`AA CC xx`** 帧，收发说明见 **§9**。
- **`UART4`（蓝牙）**：
  - 测距：`l1 = ...cm`、`l2 = ...cm`（格式同前）。
  - 灰度 **10 Hz 看门狗判离线**（连续 `Ping` 失败达阈值）时，**仅在线→离线瞬间**发送一行：`\r\nGRAY:OFF\r\n`。
  - **TEST6**（`AA BB D8`）：见 **§7.1**（`HAL_UART_Transmit` → **`huart4`**）。
  - **TEST7**（`AA BB D9`，见 **§7.2**）；**TEST1-GYCK**（`AA BB DB` 或 **PD4**，见 **§7.3**）。

### 7.1 TEST6（`AA BB D8`，灰度 / I2C 验证）

- **用途**：验证灰度**热插拔**后 **`s_gray_sensor_inited`（看门狗）、`GwGray_Ping`、数字量读、`hi2c1` 状态**是否一致，用于排查「拔掉再插后路口不识别」是否与 I2C 卡死有关。
- **动作**：蜂鸣一次 → 经 **`UART4`（`HAL_UART_Transmit(&huart4, …)`，`50 ms` 超时）** 先发一行横幅说明字段 → 再起动 **120 秒 × 1 Hz** 采样（与 `App_GraySensorWatchdog10Hz()` 同运行于 `App_Task_100ms`，顺序为：先看门狗再起动本测试逻辑）。
- **样例行格式**（行首 `\r\n`，与 `GRAY:OFF` / 测距下发风格一致）：
  - `\r\nT6 ms=<tick> init=<0|1> png=<0|1> dg=<0|1> st=<hh> e0=<hh> e1=<hh> di=<hh>\r\n`
  - **`init`**：`App_GraySensorWatchdog10Hz` 维护的在线标志（采样前瞬间副本）。
  - **`png` / `dg`**：本秒内依次调用 `GwGray_Ping`、`GwGray_ReadDigitalUpdate` 的成败（`1`=成功）。
  - **`st`**：`hi2c1.State` 的低 8 位（十六进制），对照 `stm32f4xx_hal_i2c.h` 中 `HAL_I2C_StateTypeDef`。
  - **`e0`**：Ping / 数字量读**之前**的 `HAL_I2C_GetError(&hi2c1)` 低 8 位；**`e1`**：**之后**的低 8 位（对照 `HAL_I2C_ERROR_*`）。
  - **`di`**：采样后 **`s_gray_sensor.digital_inv`**（感为寄存器读后缓存）。
- **再次触发**：可再次发送 `AA BB D8`；新会话重设计数与节拍（上一场未跑满 120 s 时由新触发覆盖）。
- **上电保护**：与其它任务相同，约前 **1200 ms** 内忽略 `D8`。

### 7.2 TEST7（路口上升沿蜂鸣）

- **用途**：验证「是否判到路口」：与测距段相同判据（**`GwGray_ReadAndDetectCross`**，阈值 **`APP_GRAY_CROSS_MIN_ACTIVE_BITS`**），在数字量**从非路口 → 路口**的上升沿 **`BuzzerDrv_Beep(80)`** 一次；停留在路口上不连续蜂鸣。
- **开关**：仅 **`AA BB D9`**（与 **`App_Test7_UserToggle()`** 等价）。每收到一次切换监听 **开/关**。**打开时再蜂鸣 1 次**作提示；**关闭时无额外蜂鸣**。
- **条件**：须 **`s_gray_sensor_inited != 0`**，否则不响。
- **调度**：于 **`App_Task_10ms`** 调用，顺序在 **`App_ProcessCrossDetect10ms` 之后**；与测距窗同时启用时同一周期可能对灰度多读一次 I²C。
- **透传**：USART1 ↔ UART5 透传开启时 **不解析 `D9`**；TEST7 需关闭透传或使用其它调试手段。热插拔后快速确认 I²C 可读请用 **§7.3（PD4）** 或 **TEST6（`D8`）**。

### 7.3 TEST1-GYCK（灰度可读性，PD4 / `AA BB DB`）

- **用途**：不依赖 **`s_gray_sensor_inited`** 门控，**连续执行 10 轮**：每轮先 **`GwGray_Ping`**，成功后再 **`GwGray_ReadDigitalUpdate`**；**10 轮均成功**才判定灰度 I²C 读数稳定可读。
- **反馈**：
  - **10 轮均成功**：**蜂鸣 1 次**（`BuzzerDrv_Beep(80)`）。
  - **任一轮 Ping 或读数字量失败**：**蜂鸣 2 次**（与 TEST5 失败相同的**队列间隔蜂鸣**，`s_pending_beeps = 2`）。
- **触发**：
  - **PD4**：每按一次执行一次检测（**不受 USART1 透传影响**，逻辑在透传判定之前执行）。
  - **`AA BB DB`**：与 PD4 **相同动作**（仅当 **未** 处于 USART1↔UART5 透传、串口正常解析时有效）。
- **与 TEST6 区别**：本项**无 UART 文本上报**，仅听蜂鸣次数即可；需寄存器级诊断仍用 **`AA BB D8`**。

## 8. 按键（`PD3` / `PD4` / `PD5`，`App_Task_10ms`）

- **PD3**：**步进电机锁定/解锁切换**（与 **`AA BB 0C` / `0D`** 语义一致：当前为 **失能** 则 **使能**→锁定，当前为 **使能** 则 **失能**→解锁；**`StepperMotorDrv_Enable` / `Disable` 返回 `HAL_OK`** 时更新内部状态并 **蜂鸣 1 次**，失败不翻转、不蜂鸣。未初始化步进驱动时无动作）。
- **PD4**：**TEST1-GYCK**，**连续 10 次** Ping + 读数字量（**全部成功** **1** 声、否则 **2** 声；见 **§7.3**）。
- **PD5**：底盘航向锁开关切换（成功后蜂鸣一次）。
- **上电约 `1200ms` 保护期**内按键事件无效（步进切换同样无效）。

### 注：Q4-V 与 `UART6` 握手

- **Q4-V** 请使用 **`AA BB 0B`**（或任务逻辑等价入口）。**PD3** 不再发送 **`AA CC DD`**、不再第二次按键启动 Q4-V。

## 9. USART6 — 连接 Linux（`AA CC` 协议）

本节描述 **UART6** 在固件中的**驱动方式**，以及 **`App_Task_10ms()`** 内对 **`AA CC xx`** 的**收发与语义**。与 **`USART1` 的 `AA BB XX`** 彼此独立。

### 9.1 硬件与链路参数（Cube / `MX_USART6_UART_Init`）

- **接口**：USART6，`115200`，`8N1`
- **引脚**：`PC6` = TX，`PC7` = RX（以工程 `ioc` / `MX_USART6_UART_Init` 为准）

### 9.2 底层收发（`Firmware/Drivers/debug_uart.c`）

| 方向 | 实现 |
|---|---|
| **RX** | **`HAL_UART_Receive_IT`** 单字节中断接收，写入 **`USART6` 环形缓冲**（在 **`DebugUart_Init()`** 内对 **`huart6`** 调用 **`HAL_UART_Receive_IT`** 启动）。 |
| **TX** | **`DebugUart6_Send`** → **`HAL_UART_Transmit`**（阻塞发送，超时由调用方毫秒参数指定）。 |
| **应用读取** | **`DebugUart6_ReadByte`**：`App_Task_10ms` 中轮询读出，非阻塞。 |

### 9.3 帧格式与对齐方式

- **定长 3 字节**：**`AA CC xx`**（`xx` 为命令字节）。
- 解析采用**滑动三字节日检**：每收到一字节即用「前两字节 + 当前字节」判断是否匹配某一种 `AA CC xx`，可多帧首尾相接，容错错位。

### 9.4 MCU → Linux（MCU 发出的帧）

| 帧 | 触发条件 | 超时 / 备注 |
|---|---|---|
| **`AA CC DD`** | **当前固件不发**。历史上曾由 **PD3** 发往 Linux 作握手；**PD3** 已改为步进切换（见 **§8**）。若仍需 **`DD`**，可在 **Linux** 侧定义或由后续 **`AA BB xx` 扩展** | — |
| **`AA CC 08`** | **仅 Q4-V**（`s_mode == 6`）：**第 4 步完成**后发送（状态 **`126U`**） | TX `50 ms`；**Q4-1～Q4-4** 经 **`126U` 时不发送** |
| **`AA CC 07`** | **仅 Q4-V**（`s_mode == 6`）：**第 7 步完成**后发送（状态 **`29U`**） | 同上；**Q4-1～Q4-4** 经 **`29U` 时不发送** |
| **`AA CC 00`** | **Q4-V 纯任务**末段：**0.92 m** 直行及末段稳定等待完成后，询问 Linux 下一步（状态 **`27U`**） | 仅 **`s_mode == 6`（`AA BB 0B` Q4-V）** 会发；**Q4-1～Q4-4**（`07`～`0A`，`s_mode` 14/16/17/18）末段为 **1.0 m**，在同位置直接接 TEST3，不发本帧 |
| **`AA CC 05`** | **`AA BB D7`（TEST5）**：每次触发发送一帧探测 | TX 超时 `20 ms` |
| **`AA CC 06`** | 收到 **`AA CC 06`** 时的**回声**：再发 **`AA CC 06`**（见 §9.5） | — |

### 9.5 Linux → MCU（MCU 解析并动作的帧）

| 帧 | 生效条件概要 | MCU 动作 |
|---|---|---|
| **`AA CC 06`** | 任意 | 立即 **回发 `AA CC 06`** |
| **`AA CC BB`** | 任意 | **当前固件无动作**（不蜂鸣）。历史上曾与 **PD3 + `DD`** 联动；该联动已取消 |
| **`AA CC 05`** | **TEST5** 已发探测且在 **1000 ms** 窗内收到 | 判 **连接成功** → 随后 **蜂鸣 1 次**（队列 `s_pending_beeps`）；超时或发送失败等为 **蜂鸣 2 次**（失败） |
| **`AA CC EE` / `01`～`04`** | **仅当** Q4-V 已发 **`AA CC 00`** 且处于等待决策（状态 **`28U`**，`s_q4v_uart6_wait_resp != 0`） | 读第三字节：**`EE` 三连蜂鸣停走**；**`01`～`04`** 跳转 **TEST3-4 / TEST3-1 / TEST3-3 / TEST3-2**；其它码停到空闲 |

**说明**：接收到 **`AA CC 05`** 时，代码会清零内部「UART6 等待应答」标志；除 **TEST5** 外固件不会自发周期性发 **`05`**，一般互不干扰。

### 9.6 Q4-V：对 `AA CC 00` 的决策应答（第三字节）

| 应答 **`AA CC xx`**（`xx`） | MCU 后续 |
|---|---|
| **`EE`** | 蜂鸣 **3** 次，回空闲 |
| **`01`** | 启动 **TEST3-4** |
| **`02`** | 启动 **TEST3-1** |
| **`03`** | 启动 **TEST3-3** |
| **`04`** | 启动 **TEST3-2** |
